FieldStock AI · Etapa 2 · Implementación

Login, perfiles y roles

Plan ordenado con patrones, rutas de archivo y verificación. Asume aprobada la arquitectura de la Etapa 1.

Patrones a aplicar

  • Controller → Service (convención del repo): toda lógica en el service; controllers thin (req/res + delegación).
  • Middleware composable: requireAuth y requireRole([...]) se aplican en la definición de rutas en routes/index.js. La auth se prepende a todo el router excepto /health.
  • Hook por módulo: useUsuarios, useUsuario en m9-usuarios/hooks/. useAuth existente se extiende para cargar el perfil del backend.
  • CSS Modules: un .module.css por página/componente.
  • Migrations idempotentes: IF NOT EXISTS + CREATE OR REPLACE VIEW con columnas nuevas al final (lección del PR #25).
  • Inyección de JWT centralizada en api.js: cada request hace pull del token actual de Supabase Auth y lo manda en el header Authorization.

Schema · migrations SQL

Se corren manualmente en Supabase Dashboard. Idempotentes — se pueden re-correr sin riesgo.

-- 1) Tabla de perfiles vinculada a auth.users
CREATE TABLE IF NOT EXISTS public.usuarios (
  id          UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  nombre      TEXT NOT NULL,
  telefono    TEXT,
  role        TEXT NOT NULL CHECK (role IN ('DUEÑO','ENCARGADO','OPERARIO')),
  activo      BOOLEAN NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ DEFAULT now(),
  updated_at  TIMESTAMPTZ DEFAULT now()
);

-- 2) Vista para exponer email (que vive en auth.users) junto al perfil
CREATE OR REPLACE VIEW public.usuarios_resumen AS
  SELECT u.id, u.nombre, u.telefono, u.role, u.activo,
         au.email, u.created_at, u.updated_at
    FROM public.usuarios u
    JOIN auth.users au ON au.id = u.id;

-- 3) Vínculo opcional remito ↔ user (para autocompletar tel en PDF)
ALTER TABLE public.remitos
  ADD COLUMN IF NOT EXISTS responsable_user_id UUID REFERENCES public.usuarios(id);

-- 4) Recrear remitos_resumen agregando responsable_telefono al final
--    (mantener orden de columnas existente — restricción de CREATE OR REPLACE VIEW)
CREATE OR REPLACE VIEW public.remitos_resumen AS
  SELECT r.id, r.numero, r.estado, r.obra, r.responsable, r.empresa_transporte,
    r.fecha_egreso, r.fecha_retorno, r.observacion, r.observacion_llegada,
    r.created_at, r.cliente_id, r.transporte_id,
    c.nombre AS cliente_nombre, c.direccion AS cliente_direccion,
    c.localidad AS cliente_localidad, c.provincia AS cliente_provincia,
    c.telefono AS cliente_telefono,
    t.nombre AS transporte_nombre, t.telefono AS transporte_telefono,
    t.contacto AS transporte_contacto,
    (SELECT count(*) FROM remito_items      ri WHERE ri.remito_id = r.id) AS cantidad_herramientas,
    (SELECT count(*) FROM remito_materiales rm WHERE rm.remito_id = r.id) AS cantidad_materiales,
    r.conductor,
    r.responsable_user_id,
    u.telefono AS responsable_telefono
   FROM remitos r
     LEFT JOIN clientes    c ON c.id = r.cliente_id
     LEFT JOIN transportes t ON t.id = r.transporte_id
     LEFT JOIN usuarios    u ON u.id = r.responsable_user_id;

Seed del primer dueño (después de crear las migrations)

Como auth.users solo se puede insertar via la API de Supabase, el seed tiene dos partes: crear el user desde el dashboard de Auth, y luego correr el SQL del perfil.

  1. En Supabase Dashboard → Authentication → Users → "Add user" con tu email y password.
  2. Copiar el uuid del user recién creado.
  3. Correr en el SQL Editor:
INSERT INTO public.usuarios (id, nombre, telefono, role, activo)
VALUES ('<UUID-COPIADO>', 'Renzo Campisi', '+54 ...', 'DUEÑO', true);

Riesgos puntuales

  • !auth.admin.createUser requiere service_role. El backend ya tiene esa key cargada (SUPABASE_SERVICE_KEY) — verificar antes de empezar la Fase 3.
  • !Validación de JWT. supabase.auth.getUser(token) hace un round-trip a Supabase por cada request. Para esta escala no es problema; si en el futuro la latencia molesta, se puede cachear el resultado por X segundos.
  • !El cliente Supabase del frontend ya persiste la sesión en localStorage — el JWT sobrevive a refrescos del browser. signOut la limpia.
  • !El check del rol "DUEÑO" tiene tilde. Consistencia en backend y frontend importa — usar una constante exportada desde un único lugar.
  • !Compatibilidad con tests existentes. 54 tests pasan actualmente. Los que pinchan endpoints van a romperse al exigir auth — hay que mockear el middleware en cada describe que use rutas protegidas, o exponer un helper de test.
  • ✓El QR mobile también requiere login. Quien escanea es siempre un empleado (operario / encargado / dueño), no el cliente. El cliente recibe el PDF impreso como comprobante. Esto simplifica: requireAuth se aplica a TODAS las rutas /api sin excepciones, y la página /remitos/:id/qr queda detrás del guard de login como cualquier otra.

Verificación end-to-end

  1. Aplicar migrations + seed → dueño existe.
  2. npm run dev desde la raíz → backend y frontend levantan.
  3. Abrir / sin sesión → redirige a /login.
  4. Loguear con el dueño → llega al dashboard, sidebar muestra "Dueño · Renzo" y el item "Usuarios" visible.
  5. /usuarios → crear un ENCARGADO. Password aparece en modal.
  6. Logout, loguear con el ENCARGADO → no ve item "Usuarios" en sidebar. Si fuerza la URL → ve la página 403.
  7. Crear remito como ENCARGADO → el responsable viene autocompletado con su nombre.
  8. Bajar PDF → "Tel. responsable" muestra el teléfono del encargado, no —.
  9. npm test --prefix fieldstock-backend → todos los tests verdes (con los mocks del middleware aplicados).
  10. npm run lint --prefix fieldstock-frontend → sin nuevos warnings.
  11. Escanear QR del remito desde celular logueado como operario → puede confirmar salida (cargando el conductor) y llegada. Si no hay sesión, la app redirige a /login antes de mostrar el QR scanner.
