Login, perfiles y roles

Sistema de autenticación con 3 roles, gestión de usuarios desde la app por el dueño, e integración del usuario logueado con el flujo de remitos.

El problema

Hoy la app no tiene autenticación: cualquiera que llegue al host puede operar todo. Además, el "responsable" del remito es un campo de texto libre que se reescribe cada vez, y el teléfono de contacto no existe en el modelo. Antes de seguir agregando features que dependen de "quién hace qué", necesitamos cerrar la base de usuarios y roles.

Las cajas en celeste son lo que se agrega en este PR. El backend sigue siendo el único guardián de permisos — el frontend solo oculta UI según el rol.

Decisiones de scope ya tomadas

  • 3 roles fijos: DUEÑO, ENCARGADO, OPERARIO.
  • Solo el DUEÑO crea usuarios desde una página dedicada dentro de la app. No hay sign-up público.
  • Creación de usuario: el sistema genera una password aleatoria de 8 caracteres (mayúsculas + minúsculas + números) y la muestra una sola vez al dueño en el modal de éxito. El dueño la copia y la pasa al nuevo usuario por fuera (WhatsApp, persona). Sin email/SMTP. Si el usuario la pierde, en una próxima iteración agregaremos un botón "regenerar password".
  • Operario sin asociación a cliente: ve todo lo que la empresa maneja; solo se limita su capacidad de modificar maestros.
  • Bootstrap del primer dueño: seed SQL manual aplicado una vez en Supabase Dashboard.
  • El user logueado se integra con remitos en este mismo PR: el responsable se autocompleta del perfil, y el teléfono del responsable sale en el PDF.

Componentes nuevos

Persistencia

  • Tabla usuarios en Postgres con id (FK a auth.users.id), nombre, telefono, role, activo. El email vive en auth.users; lo joineamos para mostrar.
  • Columna opcional responsable_user_id en remitos. El campo responsable texto se preserva por compatibilidad con datos viejos.

Backend

  • Middleware requireAuth: valida el JWT del header, carga el perfil del usuario, lo expone en req.user. Bloquea con 401 si no hay token o es inválido.
  • Middleware requireRole([...]): filtra por rol después de requireAuth.
  • Service + controller usuarios: listar, crear, actualizar, desactivar.
  • Aplicar requireAuth a todas las rutas /api excepto /health. Aplicar requireRole(['DUEÑO']) a las de gestión de usuarios.

Frontend

  • Página /login con form simple. Si hay sesión activa, redirige a /.
  • Página /usuarios (solo DUEÑO) — lista + crear + editar + desactivar.
  • Página /perfil — el user edita su propio nombre y teléfono.
  • Guard de rutas en AppRouter: sin user → /login; rol insuficiente → página 403.
  • Sidebar muestra el user actual (nombre, rol, avatar inicial) y un botón Logout.
  • Form de nuevo remito: campo Responsable se autocompleta del user logueado, editable solo si el rol lo permite.
  • api.js inyecta automáticamente el JWT en cada request.

Formas evaluadas

A · Todo-en-uno

Un PR grande con auth + tabla usuarios + roles + gestión + integración con remitos. El siguiente PR puede asumir que el sistema de usuarios está completo.

B · Incremental en 3 PRs

PR1 login básico, PR2 roles + gestión, PR3 integración con remitos. Más chico cada uno pero deja varios días con un estado a medio configurar.

C · Lite

Auth + roles + integración, pero los usuarios se crean en Supabase Dashboard (sin UI propia). Más rápido pero contradice la decisión de gestión desde la app.

Límites del scope · qué entra, qué no

Entra

  • ✓Login email + password, logout
  • ✓3 roles hardcoded con matriz de permisos validada en backend
  • ✓Gestión de usuarios (CRUD básico) desde la app por el DUEÑO, con password generada al crear (visible una sola vez)
  • ✓Edición de perfil propio (nombre, teléfono)
  • ✓Guards de rutas + sidebar con identidad y logout
  • ✓Responsable de remito autocompletado del user logueado
  • ✓PDF: tel del responsable se llena del perfil

Queda fuera (futuro)

  • —Reset de password por email (requiere SMTP en Supabase)
  • —Cambio de password desde el perfil
  • —Regenerar password de un usuario existente (botón futuro en /usuarios para cuando el operario pierda la suya)
  • —2FA, SSO con Google
  • —Auditoría de quién hizo cada acción (movimientos ya tienen responsable texto; el user_id se puede sumar después sin migración disruptiva)
  • —Asociar operarios a clientes específicos

Riesgos arquitectónicos

  • !Backend bypassa RLS de Supabase. Usa la service_role key, así que la validación de permisos vive en el middleware del backend, no en políticas de Postgres. El frontend solo oculta UI por ergonomía — la fuente de verdad es siempre el backend.
  • !Compatibilidad con remitos existentes. Los remitos viejos no tienen user_id; solo el campo responsable texto. Se preserva ese campo y se agrega responsable_user_id opcional para remitos nuevos.
  • !Bootstrap del primer dueño. El seed SQL es un punto manual fuera del control de la app. Hay que documentarlo en CLAUDE.md o un README para que sea reproducible al resetear la DB.
  • !Token de Supabase Auth. Se refresha solo via onAuthStateChange; mientras el AuthProvider esté montado, no debería expirar. Si en algún caso edge expira en medio de un request, el backend responde 401 y el frontend redirige a /login.
  • !El QR mobile también requiere login. Quien escanea (en salida y en llegada) es siempre un empleado de la empresa (operario / encargado / dueño) — no el cliente. El responsable del cliente recibe el PDF impreso como comprobante, no opera la app. Esto simplifica el modelo: todas las rutas son privadas, sin excepciones.
