M1 — Panel IA

Arquitectura de alto nivel · pendiente de aprobación

Problema

FieldStock ya tiene seis módulos generando datos: herramientas, obras, remitos, materiales, presupuestos y compras. Para responder preguntas transversales (¿qué obras tienen más herramientas?, ¿qué materiales se quedan sin stock este mes?, ¿quién es el encargado de tal obra?) hoy hay que navegar manualmente entre listas y hacer la cuenta a mano.

El Panel IA es el punto único de entrada para preguntar al sistema en lenguaje natural sobre cualquiera de esos datos, sin tener que abrir cada módulo.

Forma de la solución

Asistente conversacional con un LLM que tiene acceso de lectura a los datos del sistema a través de tool use (function calling). El usuario escribe una pregunta, el LLM decide qué endpoints internos consultar, lee los resultados y sintetiza una respuesta en castellano.

Bloques y responsabilidades

Bloque Responsabilidad
PanelChat frontend UI de chat efímero. Mantiene el historial en memoria de la sesión, muestra cada turno con su respuesta, indica cuando el LLM está "pensando" o "consultando datos".
panel.service backend Orquesta el ciclo agéntico: recibe la pregunta + historial, llama al LLM, ejecuta las tools que el LLM pide, devuelve el resultado al LLM, itera hasta tener respuesta final.
Tools registry backend Catálogo de funciones expuestas al LLM. Cada tool es un wrapper delgado sobre un service ya existente (herramientas, obras, materiales, remitos, presupuestos, compras). Todas read-only.
Provider adapter backend Capa fina que aísla el SDK del LLM. Hoy Gemini; mañana se puede cambiar a Claude o OpenAI sin tocar panel.service.

Decisiones tomadas

Decisión Por qué
Provider: Google Gemini Tier gratuito sustentable (1500 req/día, sin tarjeta). Soporta tool use. Anthropic ya no ofrece créditos iniciales gratis.
Modelo: gemini-2.0-flash Latencia baja, calidad suficiente para Q&A sobre datos estructurados, gratis en el tier free.
Read-only Las tools solo leen. Crear remitos, dar de baja, etc. queda para una v2 con confirmaciones explícitas.
Visión total (sin filtro por rol) App de uso interno. Los roles existen para "qué se puede hacer", no para "qué se puede ver". El asistente usa la service key del backend como el resto de la app. Filtros por rol se pueden sumar después.
Chat efímero Historial en memoria del frontend, no se persiste en Supabase. Persistencia en una v2.

Límites aceptados para esta versión

  • Sin acciones: el panel responde, no ejecuta nada que modifique datos.
  • Sin memoria entre sesiones: recargar la página borra el chat.
  • Sin filtros por rol: cualquier usuario logueado ve todo lo que el panel puede consultar.
  • Sin streaming de tokens: la respuesta llega de una sola vez (más simple; el costo en UX es ~2-5s de espera por turno).
  • Dependencia de internet: el panel no funciona offline (la app sí; el resto de los módulos siguen andando).
  • Riesgo residual de alucinación: acotado pero no cero — el LLM podría inventar datos si la tool falla o devuelve algo ambiguo. Se mitiga con prompt + manejo de errores en tools, no se elimina.

Cosas que NO entran en esta etapa

Esta etapa define qué hace el panel y la forma de la solución. El cómo concreto (rutas de archivo, contratos de cada tool, prompts exactos, manejo de errores y reintentos, esquema de logging para debugging) se acuerda en la Etapa 2, una vez aprobada esta.
