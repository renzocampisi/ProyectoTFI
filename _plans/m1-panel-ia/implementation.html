Plan de construcción sobre la arquitectura aprobada

Patrones aplicados

Patrón Dónde y por qué
Controller → Service Igual que el resto del backend: panel.controller solo recibe req/res y delega en panel.service.responder().
Adapter / Port panel/provider.js aísla el SDK de Gemini detrás de una interfaz pequeña (chat(messages, tools) → respuesta). Cambiar a Claude u OpenAI más adelante no toca panel.service.
Registry panel/tools.js exporta un array de tools, cada una con { name, description, parameters, handler }. Agregar tools nuevas es push al array, sin tocar el orquestador.
Loop agéntico acotado El orquestador itera llamar LLM → ejecutar tools → reinyectar resultados hasta que el LLM da respuesta final o se llega a MAX_ITERATIONS = 8. Sin tope, una tool que devuelve datos ambiguos puede entrar en loop infinito.
Hook por módulo usePanel encapsula el estado del chat (mensajes, estado, enviar, reset). El componente no llama a api directo.
CSS Modules Un .module.css por archivo (página + cada componente). Sin estilos inline para layout.

    participant FE as PanelPage

    U->>FE: escribe pregunta

{ mensaje, historial }
    API->>SVC: responder(pregunta, historial)
    loop hasta respuesta final o MAX_ITERATIONS
        SVC->>LLM: chat(messages, tools)
        alt LLM pide tool

            SVC->>TOOLS: ejecutar(nombre, args)
            TOOLS->>DB: query via service existente
            DB-->>TOOLS: datos
            TOOLS-->>SVC: resultado
            SVC->>SVC: append tool_result al historial
        else LLM responde

        end
    end
    SVC-->>API: { respuesta, traza }
    API-->>FE: 200 JSON
    FE-->>U: renderiza burbuja

Pasos ordenados

1. Plumbing del provider backend

  • Instalar @google/generative-ai en fieldstock-backend.
  • Agregar GEMINI_API_KEY al .env y al ejemplo .env.example.
  • Crear el adapter: encapsula el SDK y expone una función única chat({ system, messages, tools }) → { type, content } donde type es "text" o "tool_use".

fieldstock-backend/.env

2. Registry de tools backend

Cada tool es un wrapper delgado sobre un service existente. Todas read-only. Set inicial mínimo (~10 tools) suficiente para preguntas frecuentes:

  • listar_obras(estado?), obra_por_id(id)
  • listar_herramientas(estado?, categoria?), herramienta_por_id(id)
  • listar_materiales(), materiales_bajo_stock()
  • listar_remitos(obraId?, estado?, tipo?, desde?, hasta?)
  • listar_presupuestos(obraId?, estado?), presupuesto_por_id(id)
  • listar_compras(estado?, proveedorId?)

Cada handler aplica LIMIT 50 por defecto y proyecta solo los campos relevantes (no enviar timestamps de auditoría al LLM si no se piden — gasta tokens).

fieldstock-backend/src/services/panel/tools.js

3. Orquestador del ciclo agéntico backend

  • responder(pregunta, historial) arma el system prompt en castellano con la lista de tools y un par de ejemplos.
  • Loop hasta MAX_ITERATIONS = 8 o respuesta final.
  • Cada tool call que falla devuelve { error: "..." } al LLM en vez de tirar — el LLM decide si reintenta o reporta al user.
  • Devuelve { respuesta: string, traza: Array<{ tool, args, ok }> }. La traza sirve para debugging visual en el frontend (opcional).
fieldstock-backend/src/services/panel.service.js

4. Endpoint backend

  • Controller thin: valida que venga mensaje, llama a panel.service.responder, devuelve { data: { respuesta, traza } }.
  • Wire en routes/index.js: POST /api/panel/chat detrás de requireAuth (igual que el resto).
  • Sin requireRole — cualquier rol logueado puede usar el panel.

5. Service del frontend frontend

  • Wrapper sobre api.post('/panel/chat', { mensaje, historial }).
  • Define el shape de Mensaje y Turno en comentarios JSDoc.

6. Hook usePanel frontend

  • Estado: mensajes (array), estado (idle / enviando / error), error.
  • Acción enviar(texto): optimistic append del turno del user, llama al service, append de la respuesta o seteo de error.
  • Acción resetChat(): limpia el array.
fieldstock-frontend/src/modules/m1-panel/hooks/usePanel.js

7. UI del chat frontend

  • PanelPage: layout principal con header (título + botón "limpiar chat"), lista de mensajes con scroll automático al fondo, y composer abajo fijo.
  • ChatMessage: burbuja con avatar (user vs asistente), texto en markdown simple (negrita, listas), indicador de "pensando" cuando el último turno es del user y aún no respondió.
  • Composer: textarea + botón enviar. Enter envía, Shift+Enter newline.
  • Estado vacío inicial: sugerencias clickeables tipo "¿qué obras están activas?", "¿qué materiales tienen poco stock?", "¿qué herramientas están en mantenimiento?".

Riesgos puntuales

  • Bucle agéntico infinito: cap duro a 8 iteraciones; si llega al tope, devolver "no pude resolverlo en este turno, reformulá la pregunta".
  • Payloads grandes a contexto del LLM: cada tool limita a 50 filas y proyecta campos mínimos. Documentado en la descripción de la tool para que el LLM sepa pedir filtros.
  • API key en frontend: GEMINI_API_KEY vive solo en el .env del backend. Nunca con prefijo VITE_.
  • Rate limit del tier gratuito: 1500 req/día. Para una demo o uso interno está sobrado, pero loggear cada request en consola del backend para no quedar a ciegas si se acerca al tope.
  • Idioma de respuesta: system prompt explícitamente "respondé siempre en castellano rioplatense, conciso, sin emojis salvo que el dato lo pida".
  • Alucinación cuando una tool devuelve vacío: el handler devuelve { items: [], total: 0 } en vez de string vacío para que el LLM no infiera "no hay datos" como "no existe la entidad".
  • Latencia visible: cada turno puede tardar 3-6s. Mostrar siempre el indicador "pensando…" — sin feedback la UI se siente rota.

Verificación

  1. Levantar la app con npm run dev en la raíz.
  2. Loguearse, ir a /panel.
  3. Probar las 4 categorías de pregunta:
    • Listado simple: "¿qué obras están activas?" → debe llamar a listar_obras con estado=ACTIVA.
    • Agregación: "¿cuántas herramientas están en mantenimiento?" → llamar listar_herramientas(EN_MANTENIMIENTO) y contar.
    • Cross-módulo: "¿qué materiales se están por quedar sin stock?" → materiales_bajo_stock.
    • Fuera de alcance: "creame un remito" → debe responder educadamente que no puede ejecutar acciones todavía.
  4. Lint del frontend: npm run lint --prefix fieldstock-frontend.
  5. Probar con el segundo perfil (rol distinto) para confirmar que el endpoint responde igual (visión total intencional).
